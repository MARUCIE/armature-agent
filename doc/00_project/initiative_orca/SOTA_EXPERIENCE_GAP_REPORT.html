SOTA Gap Report

Orca CLI 体验差距报告

本报告对照 Claude Code、Codex、Amp、Kilo Code、OpenCode、GitHub Copilot 的官方资料与本机 CLI 证据，判断 Orca 当前离“SOTA 体验”还差在哪些层面，以及下一阶段应该优先补什么。

Orca 已领先 多模型协作、provider-neutral 路由、验证纪律、本机帮助冷启动
Orca 仍落后 Session 生命周期、审批/信任体系、协作与可移植性、远程/IDE/网页连续性
本机基准 orca --help 约 287.2ms，快于本机安装的 Claude/Codex/OpenCode 等入口

调研方法

  • 官方文档/官方仓库：Claude Code、Codex、Amp、Kilo Code、OpenCode、GitHub Copilot。
  • 本机 CLI 快照：claude --help、codex --help、opencode --help、gh copilot --help、Amp 原生二进制 /Users/mauricewen/.amp/bin/amp --help。
  • 本机帮助冷启动只作为运维体验参考，不当作实验室性能 benchmark。

Orca 的核心差距

P0 Session 生命周期 缺 fork / share / export / import / handoff 等一整套工作流对象能力。
P0 审批/信任操作面 虽然已有 yolo/auto/plan，但还不像 Claude Code / Codex 那样显性、系统化。
P0 搜索后只到“选中” Orca 现在已经有搜索式 picker，但还缺详情面板和 inspect-and-act 工作流。
P1 远程/网页/IDE 连续性 `serve` 还不够像完整平台能力，IDE/browser/context bridge 也偏弱。

推荐 PDCA 波次

波次目标代表动作
Wave 0统一 picker 与有限选项交互已落地：统一 picker、搜索式 picker、统一外壳
Wave 1把 session/thread 变成一等工作对象fork、share、handoff、export/import、inspect panel
Wave 2做强 trust / approval UX显式策略命令、审批详情面板、策略持久化
Wave 3把 mode 做成任务模板review/debug/ship/research/quick-assist presets
Wave 4增强远程/网页/IDE 连续性serve 升级、session portability、IDE bridge
Wave 5补性能证据面startup/TTFT/tool latency/turn cost 可视化

结论

Orca 现在最不缺的是“功能点”，最缺的是把这些能力重新组织成 operator-grade workflow surface。和 SOTA 对手相比，下一阶段最值钱的不是继续加零散命令，而是把 session 生命周期、审批/信任模型、inspect-and-act、远程/IDE/web 连续性 做成系统。
