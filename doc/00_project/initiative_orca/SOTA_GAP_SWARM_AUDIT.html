Orca CLI 的下一道门槛，不是更多模式，而是可信的异步执行

2026-04-29 的蜂群审计显示：Orca 已经具备多模型、工作流和会话连续性基础；真正的差距集中在统一执行契约、队列、证据台和信任边界。

结论

Orca 的短板已经从“能不能像主流 coding agent 一样工作”，变成“能不能让长时间、异步、多端、多代理的工作被可靠地排队、接管、审计和回放”。本轮先处理最高风险信任边界，并补上只读队列可观测面。

本轮蜂群分工

Architecture审计 chat、run、serve、mission、planner 是否共享执行契约。
Security审计 repo-local hooks、网络工具、明文会话和工具产物。
UX/TUI审计命令发现、权限提示、diff 预览、证据控制台。
Verification审计 CI、eval gate、bench 与真实发布守门的一致性。

最高优先级发现

1. Repo-local hooks 原本可以在启动时自动执行

这是本轮最尖锐的风险：仓库内 .orca / .claude hook 配置可在 REPL 启动时执行 shell，并继承进程环境。修复后，项目 hook 必须显式信任：HookManager({ trustProjectHooks: true }) 或 ORCA_TRUST_PROJECT_HOOKS=1。

2. 网络工具需要进入审批与私网保护

fetch_url 和 web_search 已纳入危险工具集合；fetch_url 现在只接受 HTTP(S)，并默认阻断 loopback、私网、link-local、CGNAT、benchmark 与常见 IPv6 本地网段。

3. 队列对象有数据基础，但 CLI 可观测性不足

本轮新增 orca queue、orca queue list、orca queue show，并在延续轮次补上 orca queue follow，把现有 TaskRun 状态与证据流变成一等操作员入口。它还不是调度器，也不是 lease manager；这是下一阶段的切入点。

4. 执行路径仍需统一

run 已经写入 WorkSession/TaskRun，但 serve 更像 prompt proxy，mission/planner 仍有各自编排状态。下一阶段应把所有入口统一写入同一执行记录。
